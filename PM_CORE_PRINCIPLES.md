# PM Core Principles

This file is the compact operating memory for Steven's 51 project-management role.
Keep it short. Use it as the source of daily reminders, not as a full conversation dump.

## Positioning

Steven is not only a meeting coordinator. The role is to turn a loose, profitable,
opaque engineering process into a visible, trackable, predictable delivery system.

Core value to leadership:

- Make real project status visible.
- Surface risks early.
- Convert Jira, Telegram, and Lark data into daily management insight.
- Reduce manual reporting with AI workflows.
- Build the operating model for a future project-management department.

## Operating Principles

1. Facts before opinions.
   Use Jira, Lark tables, Telegram discussions, and daily sync outcomes as evidence.

2. Transparency before control.
   First build dashboards and reports. Do not start with personal KPI pressure.

3. Risks must have owner, impact, deadline, and next action.
   A risk without these four fields is still vague.

4. Meetings are for clearing blockers.
   Every meeting should answer: what changed, what is blocked, who owns it, when it closes.

5. AI handles collection and first-pass analysis.
   Steven handles judgment, priority, escalation, and trust-building.

6. Do not replace the tech lead.
   Complement the tech lead by managing delivery rhythm, visibility, and cross-role coordination.

7. Show value in leadership language.
   Say less "I coordinated" and more "risk became visible, ownership became clear, delivery became predictable."

## Daily Management Outputs

The daily report should include:

- Today's Jira movements: new, resolved, updated, stale, high-priority.
- Developer action summary: who moved what, who has backlog, who has high-risk items.
- Telegram leadership signals: Ethan or key leaders' concerns, decisions, and asks.
- Current top risks: impact, owner, deadline, required support.
- Steven's next actions: what to follow up tomorrow.

## AI Workflow Backlog

Priority 1: Jira automation

- Sync all Jira issues: bugs, requirements, tasks, change items.
- Upsert by Issue key into Lark.
- Track daily added, resolved, updated, stale, and high-priority items.
- Generate daily developer action report.

Priority 2: Telegram automation

- Keep syncing fiveone-overall messages.
- Extract leader concerns, commitments, blockers, and unresolved items.
- Send daily summary to Codex thread and Telegram Saved Messages.

Priority 3: Lark dashboard

- Maintain total issue table.
- Maintain daily change table.
- Maintain developer action table.
- Maintain risk pool.
- Maintain weekly leadership summary.

Priority 4: Technical map

- Request read-only Git access, test environment access, module owner map, deployment notes.
- Use AI to understand modules and risks, not to pretend to be the tech lead.

## Upward Management

Trust has three layers:

- Information trust: reports are timely, accurate, and useful.
- Execution trust: leader asks are tracked and not dropped.
- Alignment trust: Steven is visibly helping leadership control delivery risk.

Practical actions:

- Send concise daily insight, not long raw logs.
- Maintain a separate leader-concern list.
- Ask for feedback every 1-2 weeks: what is useful, what should be reduced, what needs more focus.
- Use natural moments for relationship building: milestone, release, visit, meal, small local gift.
- Keep gifts light and non-burdensome. Work value comes first.

## Daily Reminder

Ask:

1. Did I make today's real project status visible?
2. Did I identify risks before they became surprises?
3. Did every important issue have an owner and next action?
4. Did I reduce manual work through AI or data flow?
5. Did my output help leadership see my value?
